# DN-3: Domain service signatures — whole aggregates/read models, not extracted scalars

Short, ADR-style record of a design decision — part of the Design Notes series (see [`README.md`](README.md) for the index of all notes).

**Raised:** while writing `08_guest_service_domain_services.md`'s `TableSelectionPolicy` — the obvious signature would take just `groupSize: int` (all today's rule actually reads off `GuestGroup`), but the design instead takes the whole `GuestGroup`. The same question recurred independently for `BillClosingEligibility` (same file), `OrderReadinessCheck`/`TaskSelectionPolicy` (`08_kitchen_domain_services.md`), `ActiveWaiterGuard`/`LastTableGuard`/`WaiterTerminationCompletionCheck` (`08_resource_management_domain_services.md`), and `OpenPizzeriaEligibility`/`AutoCloseEligibility` (`08_pizzeria_lifecycle_domain_services.md`) — each time resolved the same way, but never written down as its own rule, only cross-referenced as "this series' convention."

**Question:** when a domain service (a policy or a cross-aggregate guard) needs only a few fields off an aggregate or read model, should its signature take those fields directly, or the whole object?

**Decision:** take the whole aggregate or read model, even when today's implementation only reads a subset of its fields. Applies to every domain service in this series that sits behind a **policy** the domain explicitly treats as swappable (`02_discover_process_level.md` names several this way, e.g. table selection, wait-time estimation, task pickup ordering) — and, by extension, to fixed cross-aggregate **guards** too (`BillClosingEligibility`, `ActiveWaiterGuard`, and others aren't framed as swappable in `02`, but take whole objects for the same forward-compatibility reason). The only things extracted as bare scalars are true identifiers used purely for lookup (e.g. `chefId`, `waiterId` in a guard signature) — never a business fact a future variant of the rule might need.

**Rationale:**
* A policy's whole point is that it might be replaced by a different implementation later without changing its caller. If the signature only carries the fields today's implementation happens to read, a future variant that needs one more field (e.g. a VIP flag on `GuestGroup`, or per-line topping counts on `KitchenOrder`) forces a signature change — and every caller and every other implementation of that interface — instead of just a new implementation of the existing interface.
* This mirrors a discipline already used elsewhere in this series for a different reason: `08_pizzeria_lifecycle_domain_services.md`'s `OpenPizzeriaEligibility` takes the whole Readiness read model rather than two pre-extracted booleans; `08_kitchen_domain_services.md`'s `OrderReadinessCheck` takes the whole `KitchenOrder` rather than a pre-extracted `pizzaCount`. Neither of those services is a "swappable policy" in `02`'s language, but the same reasoning — don't foreclose a future variant by pre-extracting today's minimum — applies regardless of whether `02` calls it a policy or a fixed rule.
* Extraction of the read model's own filtering (e.g. **Available Tables** already being parameterised by group size before `TableSelectionPolicy` ever runs, `08_guest_service_domain_services.md`) is a separate concern and stays outside this rule — a domain service still receives *already-relevant* records, just not *pre-flattened* ones.

**Status:** Accepted, 2026-07-24.
