# DN-4: Structural dependency diagrams never show two arrows between the same pair

Short, ADR-style record of a notational/modelling decision — part of the Design Notes series (see [`README.md`](README.md) for the index of all notes).

**Raised:** while drafting `03_decompose_subdomains.md` §4's subdomain map — Pizzeria Lifecycle's `OpenPizzeria` guard genuinely needs data from Table/Waiter/Chef Management (at least one table with an `Active`-waiter assignment, at least one `Active` chef), which would mean drawing an edge back into TM/WM/CM on top of the existing TM/WM/CM → PL edges (their `Closed`-only and last-active guards), producing a two-way arrow pair. The same situation recurred repeatedly in `07_define_context_map.md` §3/§4, once per Bounded Context pair with genuine bidirectional traffic: Guest Service ↔ Pizzeria Lifecycle (status vs. visit counts), Resource Management ↔ Pizzeria Lifecycle (status vs. readiness data), Guest Service ↔ Resource Management (table/menu definitions vs. table occupancy), and Kitchen ↔ Resource Management (chef/menu definitions vs. `ChefFinishedPizza`).

**Question:** when two contexts (or subdomains) genuinely depend on each other for different data, going in different directions, should the structural dependency diagram show both edges?

**Decision:** no — a diagram never shows two arrows between the same pair of nodes, even when the reverse dependency is real. Draw only the *primary* relationship as an edge (per `07_define_context_map.md` §2: whichever side's model the other conforms to, or — where neither conforms to the other because the two facts are unrelated in kind, e.g. `03` §4's TM/WM/CM → PL edges — the pervasive/continuous need rather than the narrow/one-off one). Document the reverse dependency as prose next to the diagram ("reverse traffic" in `07`'s Relationships sections; the "deliberately not shown above" callout in `03` §4) instead of as a second edge.

**Rationale:**
* A directed edge between the same two nodes in both directions reads as a cycle, even when it isn't really one — the two edges represent two independent, differently-shaped dependencies (e.g. Resource Management depends on Pizzeria Lifecycle's status *continuously*, every guarded command; Pizzeria Lifecycle depends on Resource Management's readiness facts *once*, only when a Manager attempts `OpenPizzeria`). Showing both as identical arrows loses that distinction and makes the diagram look architecturally cyclic when the actual coupling isn't.
* `07_define_context_map.md` §2 already establishes that upstream/downstream is evaluated per kind of data, not per context pair — a context can be upstream for one fact and downstream for another, between the same two contexts. The diagram-level rule here just follows from taking that seriously: since "the relationship" isn't really a single thing between two contexts, picking one arrow to draw and pushing the rest to prose is more honest than forcing a single diagram edge to represent two unrelated relationships.
* Consistent with the project's explicit "no cycles between domains" preference: keeping every structural diagram in this series acyclic makes dependency order legible at a glance (e.g. "what must exist before X can be built/tested"), which a same-pair back-edge would immediately undermine even when technically justified.

**Status:** Accepted, 2026-07-24.
